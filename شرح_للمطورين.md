# شرح_للمطورين — مشروع contracting-site (مستند مطوّر تفصيلي)

## 1) الفكرة (Vision)
هذا النظام هدفه بناء **موقع لشركة مقاولات** بحيث تستطيع **أي شركة مقاولات** تخصيص الموقع بالكامل من **لوحة التحكم** بدون تعديل كود:
- **هوية بصرية**: شعار/فافيكون/ألوان (Primary/Secondary/Accent) + قائمة الموقع.
- **محتوى**: صفحات ثابتة، خدمات، مشاريع، أخبار، مناقصات، وظائف، عملاء.
- **تجميع طلبات الزوار**: تواصل عام + طلب خدمة + طلب توظيف (CV) + عرض مناقصة (ملف عرض).
- **أخبار تلقائية**: عند نشر/تحديث كيانات محددة يتم إنشاء خبر تلقائي ينعكس في صفحة الأخبار/الرئيسية.

الفلسفة الأساسية:
- **Admin** هو مصدر الحقيقة (Source of Truth) لمحتوى الموقع.
- **Site** يعرض فقط المحتوى “المنشور/النشط”.
- **contacts** صندوق موحّد لكل طلبات الزوار لتسهيل المتابعة.

---

## 2) الصورة العامة للنظام (Big Picture)
### مكوّنات النظام
- **واجهة الموقع (Site)**:
  - Controllers: `app/Http/Controllers/SiteController.php`
  - Views: `resources/views/site/*`
- **لوحة التحكم (Admin)**:
  - Controllers: `app/Http/Controllers/Admin/*`
  - Views: `resources/views/admin/*`
- **قاعدة البيانات**:
  - محتوى: `services`, `projects`, `pages`, `news`, `tenders`, `job_posts`, `clients`
  - إعدادات: `settings`
  - طلبات: `contacts`
  - مصادقة/نظام: `users`, `notifications`, (sessions/password_reset_tokens)
- **وسائط (Media)**:
  - تخزين الملفات: `storage/app/public/*`
  - عرض الملفات: `GET /media/{path}` (بدون `storage:link`)
- **نظام الأحداث (Events/Listeners)**:
  - إشعارات الإدارة عند وصول طلبات (`ContactRequestSubmitted`)
  - أخبار تلقائية عند حفظ محتوى (`*SavedForNews`)

### لماذا `GET /media/{path}`؟
على Windows/XAMPP شائع أن `/storage/...` يعطي 404 إذا لم يُنشأ رابط `storage:link` أو بسبب إعدادات Apache.
لهذا تم توحيد العرض عبر:
- `routes/web.php` → `GET /media/{path}` → `Storage::disk('public')->response($path)`

## 3) دورة الطلب في Laravel داخل هذا المشروع (Request Lifecycle)
هذا القسم يشرح “ماذا يحدث” من لحظة دخول المستخدم رابط حتى يرى صفحة/يتم حفظ بيانات.

### 3.1) في الواجهة (Site)
1) المتصفح يطلب URL مثل `/services/slug`
2) Laravel يطابق Route في `routes/web.php`
3) يتم استدعاء Controller method في `SiteController`
4) الـ Controller يجلب بيانات من Models (Eloquent)
5) قبل عرض View: يتم حقن `$siteSettings`, `$siteMenu`, `$sitePages` عبر:
   - `App\Providers\AppServiceProvider::boot()` → `View::composer('site.*', ...)`
6) يتم إرجاع View من `resources/views/site/...` إلى المتصفح

### 3.2) في لوحة التحكم (Admin)
1) المسؤول يفتح `/admin/...`
2) غالبًا يمر عبر Middleware `auth` (تسجيل دخول)
3) Admin Controller ينفّذ CRUD على Models
4) يتم redirect مع flash message

### 3.3) في نماذج POST (طلبات الزوار)
1) المتصفح يرسل POST (تواصل/خدمة/توظيف/مناقصة)
2) Controller: `ContactRequestController`
3) يتم validation ثم `Contact::create(...)`
4) يتم إطلاق Event: `ContactRequestSubmitted`
5) Listener: `SendAdminContactNotification`
6) تظهر البيانات في `/admin/contacts`

## 5) كيف تتدفق البيانات بين وحدات النظام (Data Flow Summary)
1) **لوحة التحكم** هي التي تكتب البيانات الأساسية (خدمات/مشاريع/أخبار/…).
2) واجهة الموقع تعرض فقط:
   - المنشور (`is_published = true`) أو النشط (`is_active = true`) حسب الكيان.
3) **الطلبات الواردة** من المستخدمين (تواصل/توظيف/مناقصة) تُخزن في جدول واحد `contacts` لسهولة الإدارة.
4) الملفات تُخزن في `storage/app/public/*` وتُعرض عبر **`/media/{path}`** لتجنب مشاكل `storage:link` على XAMPP.

## 6) Models — شرح عملي (ماذا تمثل + لماذا مهمّة)
هنا نربط بين: **الجدول** ↔ **Model** ↔ **العلاقات** ↔ **التأثير على الواجهة**.

### `App\Models\Setting`
- مسؤول عن مفاتيح الإعدادات `key/value/type`.
- `parseValue()` يحول القيم حسب النوع (خصوصًا الصور و JSON).

### `App\Models\Service`
- يمثل خدمة.
- `scopePublished()` يعيد الخدمات المنشورة بالترتيب.
- `getImageUrlAttribute()` يعرض الصورة عبر `/media`.
- علاقة `projects()` (خدمة ↔ مشاريع).
- علاقة `news()` (خدمة ↔ أخبار تلقائية عبر polymorphic).

نقطة مهمة للمطور:
- عند الحفظ في الإدارة يتم إطلاق Event لإنشاء خبر تلقائي (إن كانت منشورة).

### `App\Models\Project`
- يمثل مشروع.
- علاقة `service()` + `client()`.
- علاقة `news()` + `tenders()`.
- عند الحذف: يحذف أخبار المشروع تلقائيًا (داخل `booted()`).

### `App\Models\Tender`
- يمثل مناقصة.
- `scopePublished()` + `scopeOpen()` + `isOpen()`.
- علاقة `news()` + `project()`.
- عند الحذف: يحذف أخبار المناقصة تلقائيًا.

### `App\Models\Job`
- مربوط بجدول `job_posts`.
- `scopeActive()` + `scopePublished()`.
- علاقة `news()`.
- عند الحذف: يحذف أخبار الوظيفة تلقائيًا.

### `App\Models\News`
- خبر (يدوي أو تلقائي).
- `newsable()` polymorphic.
- `scopePublished()` يعتمد `is_published` + `published_at <= now()`.

### `App\Models\Client`
- عميل مرتبط بمشاريع.
- `getLogoUrlAttribute()` عبر `/media`.
- `projects()` + `scopePublished()`.

### `App\Models\Contact`
- طلب وارد من الواجهة.
- `cv_file_url` يعرض الرابط عبر `/media`.
- `request_type_label` يعرض نوع الطلب بالعربي (مع دعم قيم قديمة).

## 7) Routes → Controllers → Models → Views (شرح تفصيلي لوحدة وحدة)
هذا القسم يشرح “كل عملية” بشكل متدرج: من العام إلى التفصيل.

### 7.1) إعدادات الموقع (Branding / About)
#### (أ) Branding
- **Admin URL**: `/admin/settings/branding`
- **Controller**: `Admin\SettingController@branding` (عرض) و `@saveBranding` (حفظ)
- **Model**: `Setting::setValue(...)`
- **Site Consumption**:
  - `AppServiceProvider` يجلب `settings` ويوفرها كـ `$siteSettings` لكل `site.*`
  - `resources/views/site/layouts/app.blade.php` يقرأ:
    - `theme_primary_color`, `theme_secondary_color`, `theme_accent_color`
  - ثم يحقن CSS variables لتغيّر لون الواجهة مباشرة

#### (ب) About page settings
- **Admin URL**: `/admin/settings/about`
- **Controller**: `SettingController@aboutPage` و `@saveAboutPage`
- **Keys**:
  - `about_title`, `about_text_1`, `about_text_2`, `about_main_image`
- **View**:
  - `resources/views/site/about.blade.php`

### 7.2) الصفحات (Pages)
- **Admin**:
  - `/admin/pages/create` (إنشاء)
  - `/admin/pages/{page}/edit` (تعديل)
  - Controller: `Admin\PageController`
  - Model: `Page`
- **Site**:
  - `GET /pages/{slug}` → `SiteController@page` → `resources/views/site/page.blade.php`
- **التحكم بالنشر**:
  - فقط الصفحات `is_published = true` تظهر في الواجهة.

### 1) عرض صفحات الواجهة (GET)
المسارات في `routes/web.php` تتجه غالبًا إلى `SiteController`.
مثال سريع:
- `GET /services` → `SiteController@services` → `Service::published()` → `resources/views/site/services.blade.php`
- `GET /services/{slug}` → `SiteController@serviceDetails` → جلب خدمة + مشاريع مرتبطة → `resources/views/site/sub-pages/service-details.blade.php`
- `GET /projects/{slug}` → `SiteController@projectDetails` → `resources/views/site/sub-pages/project-details.blade.php`
- `GET /news` → `SiteController@news` → `News::published()->paginate()` → `resources/views/site/news.blade.php`
- `GET /news/{slug}` → `SiteController@newsDetails` → `resources/views/site/sub-pages/news-details.blade.php`
- `GET /tenders` → `SiteController@tenders` → `resources/views/site/tenders.blade.php`
- `GET /tenders/request/{tenderId}` → `SiteController@tenderRequest` → `resources/views/site/sub-pages/tender-request.blade.php`
- `GET /pages/{slug}` → `SiteController@page` → `resources/views/site/page.blade.php`

**معلومة مهمة عن الإعدادات في الواجهة**
- أي View تبدأ بـ `site.*` تحصل تلقائيًا على:
  - `$siteSettings` و `$siteMenu` و `$sitePages`
  عبر `View::composer('site.*', ...)` في `AppServiceProvider`.

### 7.3) إدارة المحتوى (Admin CRUD) — نمط موحّد
مسارات الإدارة تحت `/admin` (داخل `routes/web.php`) وتتطلب تسجيل دخول.

النمط المتكرر في كل CRUD:
1) Form في Blade (admin)
2) POST/PUT إلى Route
3) Controller:
   - validation
   - normalize translatables (إن وجدت)
   - رفع ملفات (إن وجدت)
   - create/update/delete
4) redirect + flash message
5) Site يعكس التغييرات حسب حالة النشر/النشاط

### 7.4) الطلبات من الموقع إلى قاعدة البيانات ثم إلى لوحة التحكم (Public Forms → contacts)
هذا هو “تدفق الطلبات” الذي يحتاجه المطور لفهم الدعم الفني:

#### (أ) تواصل عام
1) المستخدم يرسل نموذج التواصل `POST /contact`
2) `ContactRequestController@storeGeneral`
3) `Contact::create(...)` (request_type عربي)
4) `event(new ContactRequestSubmitted($contact))`
5) `SendAdminContactNotification` ينشئ إشعارًا للمستخدمين
6) تظهر في لوحة التحكم: `GET /admin/contacts`

#### (ب) طلب خدمة
1) من صفحة خدمة: `POST /services/{service}/request`
2) `ContactRequestController@storeServiceRequest`
3) `contacts.request_type = "طلب خدمة"`, و `service_requested = عنوان الخدمة`
4) إشعار الإدارة + ظهور في `/admin/contacts`

#### (ج) طلب توظيف + رفع CV
1) من صفحة الوظيفة: `POST /careers/{job}/apply`
2) `ContactRequestController@storeJobApplication`
3) رفع PDF إلى `storage/app/public/cv-files/...`
4) تخزين مساره في `contacts.cv_file`
5) في لوحة التحكم: زر التحميل يستخدم `/media/{path}` لضمان عدم ظهور 404 على Apache.

#### (د) عرض مناقصة + رفع ملف عرض
1) من صفحة المناقصة: `POST /tenders/{tender}/request`
2) `ContactRequestController@storeTenderRequest`
3) رفع الملف (اختياري) إلى `storage/app/public/tender-proposals/...`
4) `contacts.request_type = "عرض طلب"`
5) يظهر في `/admin/contacts` كـ “عرض طلب”.

## 8) الأخبار التلقائية (Auto News) — شرح أعمق
الغرض: عند نشر عناصر من إدارة المحتوى يتم توليد خبر تلقائي بدون إدخال يدوي.

المكوّنات:
- Events:
  - `ProjectSavedForNews`, `TenderSavedForNews`, `JobSavedForNews`, `ServiceSavedForNews`
- Listeners:
  - `SyncAutoNewsFromProject`, `SyncAutoNewsFromTender`, `SyncAutoNewsFromJob`, `SyncAutoNewsFromService`
- Service:
  - `NewsAutomationService` (منطق إنشاء/تحديث/حذف الخبر في جدول `news`)

التدفق (مثال خدمة):
1) Admin يحفظ خدمة في `Admin\ServiceController`
2) يتم إطلاق `event(new ServiceSavedForNews($service))`
3) Listener يستدعي `NewsAutomationService@syncFromService($service)`
4) يتم إنشاء/تحديث خبر في `news` (أو حذفه إذا أُلغيت عملية النشر)

ملاحظات تنفيذية:
- الخبر التلقائي مرتبط بـ:
  - `newsable_type = $model->getMorphClass()`
  - `newsable_id = $model->id`
- إن أُلغي نشر الكيان: يتم حذف الخبر المقابل كي لا تبقى أخبار “يتيمة”.

## 9) الوسائط والملفات (Media & Files) — شرح تشغيل/تشخيص
### 9.1) أين تُحفظ الملفات؟
- التخزين يتم عبر Laravel Storage Disk `public`:
  - المسار الحقيقي: `storage/app/public`
  - أمثلة مجلدات:
    - `services/` صور الخدمات
    - `projects/` صور المشاريع
    - `news/` صور الأخبار
    - `settings/` صور الإعدادات (شعار/فافيكون/…)
    - `cv-files/` ملفات السيرة الذاتية (PDF)
    - `tender-proposals/` ملفات عروض المناقصات (PDF)

### 9.2) كيف تُعرض الملفات؟
- عبر Route:
  - `GET /media/{path}` في `routes/web.php`
  - يتحقق من وجود الملف ثم يعيده كـ response

### 9.3) لماذا هذا التصميم مهم؟
- يزيل اعتماد المشروع على `public/storage` symlink.
- يقلل مشاكل 404 على XAMPP.
- يجعل كل روابط الصور/الملفات تعمل بنفس المنطق في الإدارة والواجهة.

## 10) خريطة التعديل للمطور (Where to change what)
- **ألوان الموقع/شعارات/قائمة**: `/admin/settings/branding` → `SettingController@saveBranding` → جدول `settings`
- **من نحن (نصوص + صورة)**: `/admin/settings/about` → `SettingController@saveAboutPage`
- **المحتوى**:
  - الصفحات: `/admin/pages/*`
  - الخدمات: `/admin/services/*`
  - المشاريع: `/admin/projects/*`
  - الأخبار: `/admin/news/*`
  - المناقصات: `/admin/tenders/*`
  - الوظائف: `/admin/jobs/*`
  - العملاء: `/admin/clients/*`
- **الطلبات الواردة**: `/admin/contacts` (عرض/بحث/تحديث حالة/حذف)

## 12) ملاحظات تطويرية (Developer Notes)
### 12.1) HTML من المحرر
بعض الحقول تُكتب عبر Summernote وتُخزن كـ HTML، لذلك:
- في الواجهة قد يتم عرضها بـ `{!! ... !!}` عند اكتشاف أنها HTML
- أو تحويلها إلى نص آمن عند عدم وجود HTML

### 12.2) إضافة وحدة جديدة (نمط مقترح)
إذا أردت إضافة قسم جديد قابل للإدارة:
1) Migration لجدول جديد
2) Model + Scopes (`published`, …)
3) Admin Controller + Views CRUD
4) Site routes + SiteController method + View
5) إن كان هناك ملفات: استخدم Disk `public` واعرض عبر `/media/{path}`

### 12.3) تدفق الطلبات (الأهم للدعم الفني)
أي طلب من الواجهة يجب أن:
- يمر validation واضح
- يُخزن في `contacts`
- يطلق `ContactRequestSubmitted`
- يظهر فورًا في `/admin/contacts`
